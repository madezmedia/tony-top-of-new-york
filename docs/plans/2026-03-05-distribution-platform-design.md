## Part 1 — Client-Facing Proposal

### What We're Building

T.O.N.Y. is ready to reach audiences beyond the website. This plan delivers two things:

1. **Apple TV Distribution** — T.O.N.Y. appears in Apple TV's "Watch Now" tab by syndicating the show through Apple's approved content partners. One standardized feed powers all of them.
2. **Roku Channel** — A branded T.O.N.Y. channel on Roku (the #1 streaming device in the US), launching fast with a polished custom app as Phase 2.

Both tracks are free to viewers, monetized by ads (FAST — Free, Ad-Supported Streaming). This is the same model as Tubi, Pluto TV, and Peacock's free tier. Maximum reach. No subscription barrier.

---

### How It Works

**One feed, many destinations.**

We build a single content feed — a live, auto-updating list of T.O.N.Y. episodes — that gets submitted to Apple's aggregator network and Roku simultaneously. When you add a new episode to the website, it automatically appears everywhere.

```
T.O.N.Y. Episode Library (Mux + Supabase)
            │
            ▼
     Content Feed (MRSS)
     tonyseries.com/api/feed.xml
            │
     ┌──────┴──────┐
     ▼             ▼
Apple TV        Roku Channel
Watch Now       (Phase 1 → Phase 2)
```

- Open Apple TV → Browse "Watch Now" → Find T.O.N.Y. → Watch free with ads
- Open Roku → Search T.O.N.Y. → Launch the T.O.N.Y. channel → Watch free with ads

The website's existing paywall (buy episodes early/directly) stays untouched — that's a separate track for direct supporters.

---

### Content Security

T.O.N.Y.'s video files are protected at two levels:

- **FAST distribution** (Apple/Roku): Videos are publicly accessible via Mux's CDN — this is intentional. FAST content is free-to-watch. Mux ensures reliable, high-quality delivery worldwide.
- **Website paywall**: Videos use Mux's signed URL system. Every play requires a cryptographically signed token generated by our server. Tokens expire after 4 hours and are domain-locked to the T.O.N.Y. website.

This means the same episode can exist freely on Roku while remaining paywalled on the website — because they use different Mux playback IDs with different security policies.

### Architecture: Dual Playback ID Model

Each Mux asset gets two playback IDs:

```
Mux Asset
├── playback_id: "abc123" (policy: "public")   → FAST/MRSS track
└── playback_id: "xyz789" (policy: "signed")   → Paywall/website track
```

Supabase `episodes` table extended with:
```sql
ALTER TABLE episodes ADD COLUMN mux_asset_id TEXT;
ALTER TABLE episodes ADD COLUMN mux_public_playback_id TEXT;   -- FAST
ALTER TABLE episodes ADD COLUMN mux_signed_playback_id TEXT;   -- Paywall
ALTER TABLE episodes ADD COLUMN duration_seconds INTEGER;
```

### MRSS Feed Endpoint

**Route:** `GET /api/feed.xml`
**Auth:** None (public)
**Cache:** `Cache-Control: public, max-age=3600`

**Logic:**
1. Query Supabase for episodes with `status = 'available'`
2. For each episode, build MRSS `<item>` with Mux public playback ID
3. Return XML with correct `Content-Type: application/rss+xml`

**MRSS Item Structure:**
```xml
<item>
  <title>S01E01 – Concrete Jungle</title>
  <description>Michael Cortez returns to the Bronx...</description>
  <pubDate>Sun, 12 Oct 2023 00:00:00 GMT</pubDate>
  <guid>tony-s01e01-concrete-jungle</guid>

  <media:content
    url="https://stream.mux.com/{publicPlaybackId}.m3u8"
    type="application/x-mpegURL"
    duration="2880"
    medium="video"
  />

  <media:thumbnail
    url="https://image.mux.com/{publicPlaybackId}/thumbnail.jpg?width=1920&height=1080"
    width="1920"
    height="1080"
  />

  <media:rating scheme="urn:v-chip">tv-ma</media:rating>
  <media:category>Drama</media:category>

  <!-- FAST ad support -->
  <media:advertisement type="vast">
    https://your-ad-server.com/vast?channel=tony&episode=s01e01
  </media:advertisement>

  <!-- Roku extensions -->
  <roku:adSupported>true</roku:adSupported>
</item>
```

**Channel-level metadata:**
```xml
<channel>
  <title>T.O.N.Y. — Top of New York</title>
  <link>https://tony-top-of-new-york.vercel.app</link>
  <description>A Bronx Crime Saga. Seven seasons. One family.</description>
  <language>en-us</language>
  <image>
    <url>https://image.mux.com/{heroPlaybackId}/thumbnail.jpg?width=1920&height=1080</url>
  </image>
</channel>
```

### Mux JWT Token System

#### Environment Variables
```bash
MUX_TOKEN_ID=...                    # Mux API key ID
MUX_TOKEN_SECRET=...                # Mux API key secret
MUX_SIGNING_KEY_ID=...              # Signing key ID (from Mux dashboard)
MUX_SIGNING_PRIVATE_KEY=...         # Base64-encoded RSA private key
MUX_PLAYBACK_RESTRICTION_ID=...     # Domain-restricted (website)
MUX_PLAYBACK_RESTRICTION_ROKU_ID=.. # allow_no_referrer=true (Roku devices)
```

#### Playback Restrictions

**Website restriction** (`MUX_PLAYBACK_RESTRICTION_ID`):
```json
{
  "referrer": {
    "allowed_domains": ["tony-top-of-new-york.vercel.app", "tonyseries.com"],
    "allow_no_referrer": false
  }
}
```

**Roku restriction** (`MUX_PLAYBACK_RESTRICTION_ROKU_ID`):
```json
{
  "referrer": {
    "allowed_domains": ["*"],
    "allow_no_referrer": true
  },
  "user_agent": {
    "allow_no_user_agent": false,
    "allow_high_risk_user_agent": false
  }
}
```

Roku devices do not send a `Referer` header, so `allow_no_referrer: true` is required. A separate restriction ID is used to avoid weakening the website's domain lock.

#### Token Generation (`/api/mux-token`)

```typescript
import Mux from '@mux/mux-node';

const mux = new Mux({
  tokenId: process.env.MUX_TOKEN_ID,
  tokenSecret: process.env.MUX_TOKEN_SECRET,
});

export async function generateVideoToken(playbackId: string, episodeDurationSeconds: number) {
  const minExpiry = 4 * 60 * 60; // 4 hours
  const expiry = Math.max(minExpiry, episodeDurationSeconds + 300); // +5min buffer

  return mux.jwt.signPlaybackId(playbackId, {
    keyId: process.env.MUX_SIGNING_KEY_ID,
    keySecret: process.env.MUX_SIGNING_PRIVATE_KEY,
    expiration: expiry,
    type: 'video',
    params: {
      playback_restriction_id: process.env.MUX_PLAYBACK_RESTRICTION_ID,
    },
  });
}

export async function generateThumbnailToken(playbackId: string) {
  return mux.jwt.signPlaybackId(playbackId, {
    keyId: process.env.MUX_SIGNING_KEY_ID,
    keySecret: process.env.MUX_SIGNING_PRIVATE_KEY,
    expiration: '1d',
    type: 'thumbnail',
  });
}
```

Tokens default to 4-hour lifetimes. For long episodes, `exp` is set to at least `episodeDuration + 5min` so playback cannot expire mid-stream.

Signed URLs take the form:
```
https://stream.mux.com/{signedPlaybackId}.m3u8?token={jwt}
https://image.mux.com/{signedPlaybackId}/thumbnail.jpg?token={thumbnailJwt}
```

#### Roku Token Endpoint (`/api/roku-token`)

Separate endpoint using `MUX_PLAYBACK_RESTRICTION_ROKU_ID`. Called by the Phase 2 SceneGraph app at playback start:

```typescript
// GET /api/roku-token?episodeId=s01e01
// Returns: { token: string, playbackId: string, expiresAt: number }
```

### Roku Phase 1 — Channel Submission

Roku has sunset its self-serve Direct Publisher tool. Channel submission now goes through:

1. **Roku Partner Portal** — submit via `developer.roku.com` as a custom channel (requires BrightScript app, even minimal)
2. **Distributor-assisted submission** — work with a Roku-certified distributor (e.g., Wurl, DistroTV) who can submit on your behalf using the MRSS feed

The MRSS feed is still the content source either way. It is not submitted directly to Roku as a self-serve channel anymore — it becomes the data source that feeds into the custom app or distributor pipeline.

**Recommendation:** Use the MRSS feed for Apple aggregators immediately. For Roku, proceed directly to Phase 2 custom app and skip the sunset Direct Publisher path.

### Roku Phase 2 — SceneGraph Custom App

**Directory:** `/roku` in the project root
**Language:** BrightScript + SceneGraph XML

**App structure:**
```
/roku
  manifest
  source/
    main.brs          # Entry point
    HomeScreen.brs    # Episode grid logic
    PlayerScreen.brs  # Mux HLS + JWT + VAST
  components/
    HomeScreen.xml    # SceneGraph layout
    EpisodeCard.xml   # Reusable card component
    PlayerScreen.xml  # Video + ad manager
  images/
    splash-hd.jpg     # 1280×720 brand splash
    icon-hd.png       # 336×210 channel icon
    icon-sd.png       # 248×140 channel icon
```

**Content API:** `GET /api/roku-feed` — returns episode list as JSON (not MRSS), optimized for Roku consumption.

**Playback flow:**
1. User selects episode
2. App calls `GET /api/roku-token?episodeId={id}`
3. Receives signed JWT + signed Mux playback ID
4. Constructs `https://stream.mux.com/{id}.m3u8?token={jwt}`
5. Loads into Roku `Video` node
6. `roAdManager` handles VAST pre-roll from ad tag URL

**Branding:** Gold (`#DAAB2D`) and deep black (`#020B13`), matching the website theme. Season/episode art from Mux thumbnails.

**Future:** If Roku adds mid-roll ad pods, the VAST integration extends without backend changes.

### Apple Aggregator Submission Process

**Feed URL:** `https://tony-top-of-new-york.vercel.app/api/feed.xml`

Apple's Watch Now ecosystem uses a **Partner Manifest** system. Each aggregator partner ingests the MRSS feed and surfaces content in Apple TV's Browse tab.

**Submission checklist per aggregator:**
- [ ] Feed URL validated (returns valid MRSS XML)
- [ ] All episodes have `media:content`, `media:thumbnail`, `media:rating`
- [ ] Show artwork: 1:1 square at minimum 1400×1400px, 16:9 hero at 1920×1080px
- [ ] VAST ad tag URL configured and tested
- [ ] Partner portal account created (press@tonyseries.com)
- [ ] Content rights confirmation (owned, no third-party music clearance issues)

Aggregator examples include Wurl, Zype, Brightcove, and others. The confirmed list and current submission requirements are gathered from each partner's portal before submission — the partner roster changes over time.
